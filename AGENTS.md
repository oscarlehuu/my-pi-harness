## Learned Corrections
- Don't create hard cross-extension imports (e.g. Foreman importing continual-learning); keep extensions decoupled and let them compose through the session transcript → AGENTS.md substrate.
- Validate token/theme usage against the real theme, not an identity/stub that accepts any value — a fake bg stub once hid an invalid-token crash that a green test suite still shipped.
- Advance persisted/index state only after success is confirmed (clean exit), so crashes or capped backlogs retry instead of silently dropping work; never pre-mark items processed before they're actually handled.
- In pi extension event handlers, call pi.*/ctx.* synchronously within the awaited handler — the context goes stale once the handler returns, so detached fire-and-forget tasks can't persist; defer real work to a subprocess if needed.
- Lazily/dynamically import host-only packages (e.g. the pi coding-agent package) inside functions so pure helper modules stay importable standalone outside pi's runtime.
- Give headless/crew subagents an explicit `tools:` allowlist; without one they inherit every tool (incl. interactive AskUserQuestion and foreman itself), and a headless AskUserQuestion call blocks forever (no answer-timeout) → silent idle. Route crew questions through a non-blocking escalation channel instead.
- When extracting delimited/fenced blocks (e.g. PLAN-JSON), don't assume the first marker bounds the block — scan every start→end candidate and return the first slice that actually parses, since task content can legitimately mention the markers themselves.
- Diagnose timeouts/failures from transcripts and logs before retrying; repeated timeouts are a real signal (e.g. budget too tight for an xhigh role doing many tool calls), not flaky infra to brute-force.
- xhigh reasoning models stall in long thinking spirals between actions; their thinking deltas reset the idle heartbeat, so only the max-timeout backstop catches a stall (idle can't). Reserve xhigh for generative roles (planner/developer) and use `high` for judges/doc/read-heavy roles, which emit verdicts more decisively.

## Learned User Preferences
- Match process weight to stakes: prefer direct edits for small/low-stakes tweaks and reserve Foreman's gated loop for larger/high-stakes work; the alignment ceremony should auto-scale down so small tasks stay near direct-edit.
- Verify claims against actual code, git history, live behavior, and ground truth (binary strings, live wire probes) rather than trusting status text, assumptions, or guessed wire formats.
- Esc should abort the operation (interrupt the turn), not return a cancelled result that gets responded to.
- Keep extensions independent so each works with or without the others; compose through a shared substrate (session transcript → AGENTS.md), not hard cross-extension imports.
- Dislikes passive status dashboards/mission control; wants minimal at-a-glance status (current stage + whether agents are actually alive) surfaced in the statusline as its own dedicated section.
- Surface the actual content changed (e.g. the learned bullets), not just a count, so it's visible what was written.
- Transient status UI should auto-dismiss (on a timer and on next prompt), not persist for the whole session.
- For health indicators keep normal silent and trouble loud: only tint for alarm states (yellow at stall, red at stuck), never tint the healthy/green state; show liveness as a "moved Xs ago" heartbeat (real movement), not a spinner/elapsed timer; decision prompts (e.g. awaiting ship) use an accent/pulse, not an alarm color.
- For verification, value judge independence and strength over count (diversity beats scale): keep both judges on the strong model (Opus), don't seat the cheap/fast model (Gemini) as a judge, and avoid same-model self-review (e.g. reviewer matching the developer's model). Gemini proved too flaky even for soft doc-er work — it under-produces in sustained tool-loops (many tool calls, almost no text, then idle-timeout) — so read+write doc work runs on Opus medium; reserve Gemini for the lightest soft tasks only.
- Fix root causes and harden known footguns (e.g. symlink-follow in install scripts) before shipping, rather than brute-force retrying, working around them, or leaving them in.
- Prefer enhancing existing primitives over building new marketing-named features (e.g. "ultracode"); favor sequential safety over parallel/swarm complexity unless tasks are clearly independent.
- Want the orchestrator to expose and verify its understanding — assumptions, non-goals, alternatives, blast radius — before spending tokens, and to surface scope drift early; for blind/company work, verify domain facts against artifacts and the team (treat the founder as a relay), and use principles (YAGNI/KISS/DRY) as self-critique lenses, not badges to stamp.

## Learned Workspace Facts
- my-pi-harness is a workspace of pi extensions (not a user-facing app); extensions live in extensions/<domain>/, register tools via pi.registerTool, jiti-loaded TS, no build step. Philosophy: "primitives, not features" — each extension stands alone yet composes.
- Extensions install via symlinks (~/.pi/agent/extensions/<name>), so file edits are live; only a pi restart is needed; ./install.sh is only for adding new extension folders.
- Installed pi version is 0.78.0; ctx.abort() interrupts the agent turn (stopReason "aborted"). /reload re-imports extensions/skills/prompts/keybindings/themes but only in the current session (other running sessions need their own reload/restart); extension code loads at startup so a reload is required for code changes, while crew .md allowlists are re-read on each subprocess spawn.
- pi main-session transcripts live at ~/.pi/agent/sessions/--<cwd>--/*.jsonl (JSONL); AGENTS.md auto-loads as context. Slug rule: strip leading slash, replace / \ : with -, wrap in --.
- pi exposes turn_end / agent_end / stop events; pi tool names are lowercase (read, grep, find, ls, bash, edit, write). pi's --tools is an allowlist covering built-in, extension, and custom tools (agent frontmatter `tools:` maps to it, so omitting a tool reliably blocks it); --exclude-tools is the denylist.
- Foreman is gated orchestration (planner→dev→test→review→ship with human gates); stages are per-round, pre-ship, release (release only supports commit); its guard treats .md paths as no-impact; commit stages only the task's filesChanged + ledger, never git add -A. Toggle direct-edit mode as the escape hatch. Concurrent sessions share one full-suite verify gate, so parallel runs can flake it red and cause false-negative escalations; an escalated task is terminal (resume just re-reports it, no re-verify path) and hands control back to the orchestrator to finish by hand. Crew agents now have explicit tools: allowlists (developer/ui-developer get read/write/edit/bash/grep/find/ls + escalate_question; planner/tester read-only) — AskUserQuestion and foreman are blocked for crew. Crew raise questions via the crew-only escalate_question tool (FOREMAN_CREW=1), which records to the ledger and returns instantly → loop pauses as awaiting_decision state → CTO answers from context or asks the founder via its own AskUserQuestion → foreman({resume:true, answer}) injects it into the next round. The tester only sees task + diff, not ledger events, so it can't tell a hardcoded value was founder-approved. Foreman's guard allows git add/commit/status but blocks git apply/restore/checkout --. Crew subprocesses run under a shared timeout harness (agent-timeouts.ts): per-role idle/max budgets (planner 180s/480s, developer/ui-developer 180s/900s, tester 90s/300s, reviewer 180s/720s, doc-er 180s/720s) overridable via FOREMAN_<ROLE>_IDLE_MS/_MAX_MS, with heartbeat + AbortController + SIGKILL backstop; on timeout tester→fail (never silent ship), reviewer→flagged/inconclusive (never auto-approve), developer→retry round, doc-er→soft-flag (never blocks). Strict DoD refuses to ship without a clean reviewer APPROVE even after founder approval (no force-ship bypass). Foreman only reuses a persisted planner draft when its source is "planner" (fallback drafts re-draft); historically a same-slug fallback draft stuck, so forcing a real planner re-run could need a cleared ledger / clean slug. Crew model routing: CTO/planner/tester/reviewer run Opus (reviewer is high, not xhigh — xhigh stalled in thinking spirals), developer runs GPT-5.5, scout/ui-developer run Gemini 3.5 Flash (id cliproxy/gemini-3.5-flash-low:high), doc-er runs Opus medium — so developer↔judges are already cross-model (decorrelated). cliproxy (cli-proxy-api v7.1.55, router-for-me/CLIProxyAPI) supports low/medium/high/xhigh thinking levels (all probed live), not just the high/xhigh declared in models.json. PLAN-JSON carries understanding/assumptions(+confidence)/nonGoals/alternatives/blastRadius surfaced at Gate 1; tester.md and reviewer.md use a default-refuted adversarial stance (must try to disprove before passing). After Gate 1 approval the founder-approved intent (understanding/assumptions/non-goals) flows down into the developer and tester prompts (re-attached every round so the fail-retry devContext rebuild doesn't drop it), so crew build/judge against one shared contract instead of re-guessing intent. A doc-er stage runs after reviewer APPROVE and before Gate 2 (soft: failures flag, never block; emits NONE when docs already cover the change), backed by a hard doc-drift detector that flags stale docs at Gate 2; doc-er writes docs/ + CHARTER (never AGENTS.md, owned by continual-learning). extensions/foreman/docs/INTERNALS.md is the agent-friendly code-map of index.ts. Adding a new crew role requires ./install.sh to symlink its agent .md into the live agent dir (else ENOENT at spawn). A scorer (scorer.ts, pure) ranks each plan assumption by risk = P(wrong) × cost and routes it self/founder/team (cost from blastRadius/paths + highRiskPaths globs in foreman.json); a team channel (teampacket.ts) emits a paste-able packet under assume-unless-vetoed (founder relays to the team, veto via the existing reject path). Anti-rubber-stamp tiers 1+2+4 ship: claims must carry evidence (else flagged unsubstantiated), the planner self-gates before asking, and shipping an elevated/high-risk change needs a confirm token (foreman resume with confirm:"SHIP <slug>") instead of one-tap — dormant until highRiskPaths is configured.
