# Foreman Internals

> This is the code-level internals map for the Foreman extension. `CHARTER.md` remains the framework/kernel contract, and repo/agent `AGENTS.md` files hold learned process/preferences from continual learning; keep those concerns separate.

## 00 - Start here

- **Changing the loop starts in `index.ts`, not the helpers.** The foreman tool is registered at `extensions/foreman/index.ts:1259`; the bounded round loop starts at `while (state.round < state.maxRounds)` in `index.ts:1582`.
- **Changing prompt/context contracts starts in pure formatters, then re-attach inside the loop.** `formatIntentContract()` lives in `planner.ts:348`; resolved escalation answers use `formatResolvedDecisions()` in `index.ts:536`; both are attached to developer prompt material inside the loop at `index.ts:1605-1610` and to `testerTask` at `index.ts:1775-1782`.
- **Changing gates starts in `.pi/foreman.json` and `gates.ts`.** Gate kinds/stages are `command|judge|action` and `per-round|pre-ship|release` (`gates.ts:13-14`); command execution is `runCommandGates()` (`gates.ts:215`).
- **Changing strict ship behavior starts in `done.ts`, then the Gate 2 approve branch in `index.ts`.** DoD evaluation is `evaluateDoneness()` (`done.ts:45`), log extraction is `extractDonenessInputs()` (`done.ts:140`), and strict Gate 2 approval blocks at `index.ts:1541-1555`.

## 01 - Maintenance format for this doc

- **Use stable section headers.** Keep the `## NN - Name` shape so a future doc-er stage can replace one section in place without reparsing prose.
- **Lead with the actionable fact.** Each bullet should tell an agent what to touch or never touch before explaining why.
- **Anchor non-obvious claims.** Prefer `file.ts:functionName` for stable navigation; use `file.ts:line` when the line is a durable seam such as `while (state.round < state.maxRounds)`.
- **Keep framework vs internals split.** Put product/process rules in `CHARTER.md`; put code layout, invariants, and extension recipes here.

## 03 - `index.ts` control flow

- **Tool registration and state resolution happen before the loop.** `execute()` starts at `index.ts:1276`; resume restores wiped ledgers and resolves the caller-owned task at `index.ts:1289-1296`; new tasks call `initLedger()` at `index.ts:1300`.
- **Gate discovery is refreshed by `refreshGates()`.** It loads `.pi/foreman.json` or legacy `verifyCommand` via `loadGates()` at `index.ts:1318`, then filters per-round command gates at `index.ts:1319`.
- **Gate 1 blocks before code.** The Gate 1 section starts at `index.ts:1394`; approval may write proposed `.pi/foreman.json` through `writeProposedManifestOnGate1Approval()` (`index.ts:1403-1420`); otherwise `draftPlannerPlan()` runs/persists plan material and emits the approval prompt (`index.ts:1429-1475`).
- **Founder-approved intent is loaded once, but attached every round.** After Gate 1, persisted plan JSON is read and rendered through `formatIntentContract()` at `index.ts:1479-1483`; the actual per-round re-attachment is at `index.ts:1605-1610`.
- **Crew decision resume is a gate-style pause before the loop.** If state is `awaiting_decision`, reject redirects at `index.ts:1497-1508`; answer appends `state.resolvedDecisions` at `index.ts:1511-1516`; no answer re-emits `formatDecisionPrompt()` at `index.ts:1525-1527`.
- **Gate 2 resume either reopens, blocks on strict DoD, ships, or re-prompts.** The Gate 2 resume branch starts at `index.ts:1530`; reject rebuilds `devContext` at `index.ts:1532-1540`; approve evaluates DoD and blocks if not done at `index.ts:1541-1555`; successful approval marks `done`, runs release action gates, and emits ship output at `index.ts:1558-1569`.
- **The bounded round loop is the loop.** `while (state.round < state.maxRounds)` at `index.ts:1582` is the only developer/tester retry loop; rounds are exhausted and escalated at `index.ts:2113-2117`.
- **Round order is developer -> per-round command gates -> tester -> pre-ship -> soft doc-er/drift -> Gate 2.** Developer starts at `index.ts:1588`, command gates at `index.ts:1752` (`runCommandGates(gates, "per-round", ...)`; the VERIFY block opens at `index.ts:1727`), tester at `index.ts:1767`, success/pre-ship decision at `index.ts:1867`, the soft doc-er stage runs through `runDocErStage()`, and Gate 2 await follows it.
- **Developer step writes a controller-owned handoff.** The developer subprocess is invoked through `runAgentWithTimeout()` at `index.ts:1610`; frontend fallback may rerun the same prompt with `UI_FALLBACK_MODEL` at `index.ts:1622-1659`; `writeHandoff()` records the developer handoff at `index.ts:1682`.
- **Crew escalation pauses immediately after the developer step.** The parent reads `pending_question.json` at `index.ts:1707`, copies it into durable state as `awaiting_decision` at `index.ts:1709-1713`, emits the decision prompt (`emit(formatDecisionPrompt(...))`) at `index.ts:1723`, then returns.
- **Per-round command gate exit code is ground truth.** `runCommandGates(gates, "per-round", ...)` runs at `index.ts:1752`; a non-zero gate forces `successState = "fail"` even if the tester would pass (`index.ts:1814-1818`).
- **Tester judges intent after command gates.** `testerTask` is built at `index.ts:1776-1783`; tester runs through `runAgentWithTimeout()` at `index.ts:1798`; tester handoff/log state are written at `index.ts:1832-1853`.
- **Pre-ship command gates reopen the developer loop on failure.** Pre-ship gates split by kind at `index.ts:1872-1875`; command gates run at `index.ts:1906`; any failure rewrites `devContext` and `continue`s at `index.ts:1914-1928`.
- **Pre-ship judge gates reopen only on `REQUEST-CHANGES`.** Reviewer agent runs at `index.ts:1960-1970`; parser/decision run at `index.ts:1985-1991`; blocking reviewer output rewrites `devContext` and continues at `index.ts:2025-2036`; unknown reviewer output is flagged, not approval, at `index.ts:2039-2054`. After clean reviewer approval (or when no reviewer gate is declared), `runDocErStage()` runs the soft documentation refresh before Gate 2.
- **Gate 2 is a pause, not a commit.** Passing tester/pre-ship first runs the soft doc-er/drift stage, then sets `state.state = "awaiting_ship"`; release actions only run later on Gate 2 approval (`index.ts:1563-1565`). Gate 2 emits the doc-er result and any `docs may be stale` drift warnings as advisory founder-visible status, not DoD blockers.
- **Fail retry rebuilds `devContext` wholesale.** Tester failure replaces `devContext` at `index.ts:2106-2110`; other replacement sites include decision redirect (`index.ts:1502`), decision answer (`index.ts:1519`), ship reject (`index.ts:1537`), implementer timeout (`index.ts:1696`), pre-ship command failure (`index.ts:1923`), and reviewer changes (`index.ts:2028`).

## 04 - Crew invocation seams and heartbeat

- **`runAgent()` is transport only.** It assembles the `pi --mode json -p --no-session` subprocess args, applies model/tool allowlists, appends the crew prompt, streams JSON events, records transcript events, and returns `{ text, exitCode, stderr }` (`index.ts:236`).
- **Quota safety lives in the transport seam.** Crew system prompts are passed with append-only `--append-system-prompt` at `index.ts:258`; do not replace this with `--system-prompt` for cliproxy/Claude agents.
- **Crew context env is injected by transport.** `FOREMAN_CREW=1` is always set for crew subprocesses at `index.ts:277`; task slug/cwd/round are passed as `FOREMAN_TASK_*` at `index.ts:279` so `escalate_question` can write the right ledger.
- **Activity heartbeat comes from JSON stream movement.** `runAgent()` calls `options.onActivity?.()` for each parsed JSON event at `index.ts:319`; `runAgentWithTimeout()` wraps this with `recordActivity()` (`index.ts:456`) to update `lastActivityAt` (`index.ts:459`) and reschedule the idle timer (`index.ts:450-453`).
- **`runAgentWithTimeout()` is the timeout wrapper.** It owns `AbortController`, idle/max timers, timeout reason, parent abort propagation, and returns both the run result and timeout outcome (`index.ts:422-476`).
- **Per-role timeout defaults live outside `index.ts`.** `AGENT_TIMEOUTS` is resolved once at `index.ts:88` from `resolveAllAgentTimeouts()`; defaults are planner 180s/480s, developer/UI 180s/900s, tester 90s/300s, reviewer 180s/720s, and doc-er 180s/720s (`agent-timeouts.ts:54-60`).
- **Timeout degradation policy is deterministic.** Planner timeout falls back to deterministic plan, developer/UI timeout retries the developer round, tester timeout fails the verdict, reviewer timeout is flagged/inconclusive, and doc-er timeout is a soft flagged documentation outcome (`agent-timeouts.ts:133`).
- **Dashboard liveness is real movement, not a spinner alone.** Transcript events include timestamps from `makeTranscriptWriter()` (`index.ts:214`); `dashboard/reader.ts` derives `lastMovementMs` from the last `tool_call.t` in `readActiveTranscriptInfo()` (`dashboard/reader.ts:485-519`).

## 05 - Founder-approved context flow

- **Gate-1 intent is a persisted planner-plan projection.** `readPersistedPlannerDraft()` reads `plan.json` at `index.ts:681`; `formatIntentContract()` renders `understanding`, `assumptions`, and `nonGoals` at `planner.ts:348-362`; `index.ts` converts it to developer/tester headers at `index.ts:1481-1483`.
- **Escalation answers are a separate persisted contract.** `ResolvedDecision` is defined in `ledger.ts:37-46`; answers append to `state.resolvedDecisions` at `index.ts:1511-1516`; `formatResolvedDecisions()` renders them as authoritative founder choices at `index.ts:536-543`.
- **Both contracts go to the developer every round.** Inside the round loop, `decisionsForDev`, `devContextWithIntent`, and `devContextWithDecisions` are rebuilt immediately before invoking the developer (`index.ts:1605-1610`).
- **Both contracts go to the tester.** `testerTask` includes `intentForTester` and `decisionsForTester` at `index.ts:1775-1783`, including explicit language that non-goals should not fail and founder-chosen literals are approved.
- **Critical invariant: re-attach per-round context inside the loop.** The fail/retry path replaces `devContext` wholesale at `index.ts:2106-2110`, and other branches do the same (`index.ts:1502`, `index.ts:1519`, `index.ts:1537`, `index.ts:1696`, `index.ts:1923`, `index.ts:2028`). If you inject new founder-approved context only into the initial `devContext` literal at `index.ts:1485`, it will disappear on retry.

## 06 - Escalation pause/resume channel

- **Crew agents never block on the founder.** The crew-only `escalate_question` tool is registered only when `FOREMAN_CREW=1` at `index.ts:1149-1150`; the developer/UI crew allowlists include `escalate_question` in `crew/developer.md:4` and `crew/ui-developer.md:4`.
- **Escalation writes a pending question file, then the crew stops.** The tool validates task env (`FOREMAN_TASK_SLUG`, `FOREMAN_TASK_CWD`, `FOREMAN_TASK_ROUND`) at `index.ts:1163-1166`, writes `pending_question.json` via `writePendingQuestion()` at `index.ts:1187`, and tells the agent to stop at `index.ts:1191-1197`.
- **Parent loop consumes pending question after developer returns.** It reads the pending question at `index.ts:1707`, clears the file at `index.ts:1709`, copies it into `state.pendingDecision`, sets state `awaiting_decision`, and writes state at `index.ts:1710-1713`.
- **Resume with `answer` folds the decision into durable state.** The `awaiting_decision` resume branch appends a `ResolvedDecision` at `index.ts:1511-1516`, so future developer rounds and the tester see the answer through `formatResolvedDecisions()`.
- **Resume with `reject` redirects instead of answering.** Redirect clears pending state and rebuilds developer context with new direction at `index.ts:1497-1508`.
- **Reusable rule:** this is the general pause/resume channel for crew-only material decisions. New crew roles should record via the same non-blocking tool instead of adding interactive UI to headless agents.

## 07 - Gate pipeline and strict DoD

- **Gate declarations are data first.** `.pi/foreman.json` gates normalize through `loadGates()` (`gates.ts:127`); supported gate kinds/stages are `command|judge|action` and `per-round|pre-ship|release` (`gates.ts:13-14`).
- **Legacy `verifyCommand` is only a fallback.** If no `.pi/foreman.json` exists, `loadGates()` synthesizes one per-round command gate named `verify`; if the manifest exists but is bad, it returns valid entries only and does not synthesize legacy fallback (`gates.ts:127-144`).
- **Command gates run in declaration order and do not short-circuit.** `runCommandGates()` iterates all command gates for a stage at `gates.ts:215-226`, and `passed` requires every gate to exit 0 (`gates.ts:228`).
- **Exit code is ground truth.** In `index.ts`, any non-zero per-round command gate forces `successState = "fail"` (`index.ts:1814-1818`); tester judgment can still fail/partial/block a passing command gate (`index.ts:1819-1823`).
- **Pre-ship has command gates and judge gates.** Pre-ship commands run at `index.ts:1890-1932`; pre-ship judge gates run reviewer-like agents at `index.ts:1936-2058`; action gates at pre-ship are skipped as unsupported at `index.ts:1875-1885`.
- **Reviewer `APPROVE` is the only clean reviewer pass.** `parseReviewVerdict()` parses the line at `reviewer.ts:77`; `decideReviewOutcome()` reopens only on `request-changes` and flags unknown at `reviewer.ts:88-108`; strict DoD still treats unknown as not done when a reviewer gate exists (`done.ts:85-93`). The doc-er stage runs only after clean reviewer approval (or no reviewer gate) and never converts unknown review into approval.
- **Gate 2 approval cannot force-ship past strict DoD.** On approval, `evaluateCurrentDoneness(true)` runs at `index.ts:1541`; if `!doneness.done`, Foreman keeps `awaiting_ship`, logs blockers, and says strict mode has no force-ship bypass (`index.ts:1544-1555`).
- **Release actions run only after Gate 2 approval and strict DoD.** Release action gates are selected at `index.ts:1563`; `runReleaseActionGates()` supports only `action: "commit"` and skips unknown actions (`index.ts:1050-1066`).
- **Commit staging is path-scoped, never whole-tree.** `runReleaseCommitGate()` derives `stagePaths` via `resolveStagePaths()` at `index.ts:986`, then runs `git add -- ...stagePaths` at `index.ts:1002`; `ship.ts` filters unsafe whole-tree pathspecs like `.`, `-A`, `--all`, `*`, and `/` at `ship.ts:110-132`, and defaults to developer `filesChanged` plus doc-er updated doc paths plus the ledger dir at `ship.ts:143-145`.

## 08 - Extension recipes

- **To inject new founder-approved context into crew:** add/extend a pure formatter in `planner.ts` (pattern: `formatIntentContract()` at `planner.ts:348`), persist/validate any new planner fields in `validatePlannerPlan()` (`planner.ts:200`), then re-attach beside `formatResolvedDecisions()` inside the round loop (`index.ts:1605-1610`) and inside `testerTask` (`index.ts:1775-1783`). Test with `extensions/foreman/test/planner_test.sh` and any flow-specific headless test.
- **To add a new timeout-guarded crew role:** add the role to `AgentTimeoutRole`/`AGENT_TIMEOUT_ROLES` and `DEFAULT_AGENT_TIMEOUTS_MS` (`agent-timeouts.ts:12`, `agent-timeouts.ts:47`, `agent-timeouts.ts:54`), add env prefix handling in `ROLE_ENV_PREFIX`/`timeoutEnvKeys()` (`agent-timeouts.ts:64-85`), extend degradation if needed (`agent-timeouts.ts:133`), then invoke it through `runAgentWithTimeout()` (`index.ts:422`) with a transcript role/dashboard plan. Pattern: doc-er uses timeout role `"doc-er"` and degrades softly to Gate 2 status, never a hard DoD blocker.
- **To add a command gate:** prefer declaring it in `.pi/foreman.json`; no code change is needed if it is `kind: "command"` with stage `per-round` or `pre-ship` (`gates.ts:13-14`, `gates.ts:215`).
- **To add a new gate kind/stage/action:** change normalization in `gates.ts` first (`normalizeGate()` around `gates.ts:102-115`), then add orchestration in `index.ts` at the relevant stage (`per-round` near `index.ts:1752`, `pre-ship` near `index.ts:1869`, `release` near `index.ts:1050`/`index.ts:1563`). Keep side-effect policy in `index.ts`; put pure decisions in helper modules.
- **To add a release action:** extend `runReleaseActionGates()` (`index.ts:1050`) and add pure helper decisions in `ship.ts` rather than embedding policy in git/process code. Preserve path-scoped staging semantics from `resolveStagePaths()` (`ship.ts:139`).
- **To add a DoD check:** log a durable event in `index.ts`, parse it in `extractDonenessInputs()` (`done.ts:140`), evaluate/render it in `evaluateDoneness()`/`renderDoneChecklist()` (`done.ts:45`, `done.ts:119`), and update `extensions/foreman/test/reviewer_test.sh` or a new `done` test if the checklist semantics change.
- **To add dashboard/statusline data:** derive it in `dashboard/reader.ts` first (`buildStatuslineModel()` at `dashboard/reader.ts:545`, `buildRootRows()` at `dashboard/reader.ts:927`), then render it in `dashboard/view.ts`. Do not make `dashboard/view.ts` authoritative for loop state.
- **To add a new machine block parser:** use or preserve the `extractJsonBlock()` scanning behavior (`index.ts:486-507`) so prose mentions of markers do not poison parsing.

## 09 - NEVER boundaries / footguns

- **NEVER replace append-only crew prompts with `--system-prompt`.** `runAgent()` uses `--append-system-prompt` at `index.ts:258`; this preserves cliproxy/Claude quota routing and avoids replacing the host prompt.
- **NEVER add pi SDK/TUI imports to pure helper modules.** The pure modules advertise this in headers (`planner.ts:4`, `gates.ts:4`, `reviewer.ts:4`, `docdrift.ts:1`, `ship.ts:4`, `done.ts:4`, `agent-timeouts.ts:4`, `dashboard/reader.ts:2`); keep side effects and host APIs in `index.ts` or `dashboard/view.ts`.
- **NEVER make headless crew agents ask interactive questions.** Use the crew-only `escalate_question` channel (`index.ts:1149-1197`) and resume through `awaiting_decision`; do not give crew `AskUserQuestion` or `foreman` access.
- **NEVER inject retry-critical context only into the initial `devContext`.** Retry branches replace `devContext` wholesale (`index.ts:2106-2110` and other sites listed in Section 05); attach new context immediately before `runAgentWithTimeout()` inside the loop (`index.ts:1605-1610`) and into `testerTask` (`index.ts:1775-1783`).
- **NEVER let a non-zero command gate become success.** `index.ts:1814-1818` makes command exit code ground truth, and `gates.ts:228` requires all command gates to pass.
- **NEVER let doc-er block ship or edit outside documentation roots.** The doc-er crew is soft: timeout/error/no-op becomes founder-visible Gate 2 status, while writes are bounded by prompt to `docs/` and `extensions/*/docs/`; it must never touch code or `AGENTS.md`.
- **NEVER treat reviewer unknown/timeout as approval.** Unknown reviewer output may proceed to Gate 2 flagged (`reviewer.ts:101-108`, `index.ts:2039-2054`), but strict DoD blocks ship until a clean `APPROVE` when a reviewer gate is declared (`done.ts:85-93`, `index.ts:1544-1555`).
- **NEVER stage the whole tree for release commits.** Commit release gates stage `stagePaths` only (`index.ts:986`, `index.ts:1002`); `resolveStagePaths()` filters whole-tree pathspecs and defaults to developer-reported files plus doc-er updated docs plus the ledger (`ship.ts:110-145`).
- **NEVER assume the first JSON marker pair is the payload.** `extractJsonBlock()` scans every start marker and returns the first slice that parses as JSON (`index.ts:486-507`); preserve this when touching `PLAN-JSON`, `DEV-JSON`, or similar contracts.
